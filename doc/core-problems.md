# takusu-core planning algorithm の問題と改善方針

## Summary

`proposal.typ` の設計思想と現在の `takusu-core` を比較して確認された問題をまとめる。

優先順位は以下とする。

1. `plan_in_range` が期間外タスクとの制約を評価しない問題を修正する
2. 不確実性バッファを実際の連続空き時間として評価する
3. 一日へのタスク集中を避ける日次負荷評価を追加する
4. 実行不可能時の task abandonment を実装する
5. 依存グラフの入力検証を追加する
6. 睡眠時間の重複加算を修正する

`RescheduleRange` の境界判定については、現在の「期間と交差するタスクを再配置対象にする」挙動を許容し、本書の対象外とする。

## 設計上の目標

`proposal.typ` が重視しているのは、単に締切違反が少ないスケジュールではなく、ユーザーが実際に従いやすく、見積りの不確実性や生活上の制約を反映したスケジュールである。

改善後も次の原則を維持する。

- 5分スロットに離散化し、ヒューリスティック探索で準最適解を求める
- 依存関係、開始可能時間、固定予定、並列条件は実行可能性に関わる制約として扱う
- 標準偏差は、タスクの実行時間が延びるリスクとして扱う
- 睡眠と一日当たりの負荷は、ユーザーがスケジュールを拒否しないための生活制約として扱う
- 全タスクが収まらない場合は、暗黙に締切超過させるだけでなく、短縮・断念した結果を明示する
- 部分再計画では、変更対象外の予定を動かさず、それらとの制約を必ず評価する

## Problem 1: `plan_in_range` が期間外タスクとの制約を評価しない

### 現状

`Planner::plan_in_range` は期間外タスクを sub-planner から除外し、期間内タスクだけを解いたあとで期間外の予定を結果へ追加している。

この構造では、探索中に期間外タスクが存在しないため、次の関係を評価できない。

- 期間内タスクと期間外タスクの時間重複
- 両者の並列実行条件
- 期間内タスクが期間外タスクに依存する関係
- 期間外タスクが期間内タスクに依存する関係

期間内タスクの `depends` は sub-planner のIDへ変換する際、期間外タスクを指す依存が削除される。そのため、期間外の依存先が終了する前に依存元が配置される可能性がある。

### 影響

ユーザーが一部の時間帯だけを再計画したとき、動かしていない予定と新しい予定が重複したり、依存順序が壊れたりする。部分再計画は日常的な調整に使われるため、スケジュールへの信頼を直接損なう。

### 改善方針

sub-planner を作らず、元の `Planner` に対して期間外タスクを pinned とした partial solve を行う。

```text
pinned = current schedule のうち期間外または明示的に pin されたエントリ
solve_partial(original_planner, pinned)
```

これにより、元のtask IDと依存グラフを維持したまま、固定・再配置タスクを同じ評価関数へ渡せる。

pinned タスク自身が再配置タスクに依存している場合、依存を満たすために pinned タスクを動かすことはできない。したがって、次のいずれかを結果として返せるようにする。

- 再配置タスクだけを動かして依存を満たせた plan
- pinned 条件下では依存を満たせないことを示す warning/error

### 必要なテスト

- 再配置タスクが pinned タスクと非並列で重複しない
- 再配置タスクが pinned 依存先の終了後に配置される
- pinned タスクが再配置タスクに依存するケースを検出する
- `extra_pinned` の位置が変わらない
- task IDと全task数が維持される

## Problem 2: 不確実性バッファが実際の空き時間を表していない

### 現状

現在の `buffer_score` は概ね次の式になっている。

```text
sigma(task) × (deadline(task) - scheduled_end(task)) × W_BUFFER
```

タスク終了後から締切までに別のタスクが入っていても、その区間全体がバッファとして評価される。このため、標準偏差の大きいタスクを早く置くだけでスコアが増え、直後が別タスクで埋まっていて実際には延長できない配置も高評価になる。

`proposal.typ` の例では、標準偏差の小さいAを先に、標準偏差の大きいBを後に置くことで、Bの延長が後続タスクへ影響しないようにしている。現在のテストは逆に、高sigmaタスクを先に置く方を期待している。

### 影響

見積りが外れやすいタスクが延長したとき、後続予定を連鎖的に壊すスケジュールが生成される。不確実性を正規分布で表す設計上の利点を十分に活用できていない。

### 改善方針

バッファを「タスク終了後に連続して利用できる空き時間」として計算する。

```text
buffer_end = min(deadline, 最初に競合する後続タスクの開始)
actual_buffer = max(0, buffer_end - scheduled_end)
buffer_score = sigma × actual_buffer × W_BUFFER
```

後続タスクとの競合判定では、`parallelizable` と `allows_parallel` を考慮する。対象タスクが延長しても合法的に並行できる後続タスクは、バッファ終端にしない。

将来的には正規分布をより直接使い、確保時間で完了できる確率や期待超過時間を評価する方法も検討する。ただし最初の修正では、連続空き時間による評価の方が単純で説明しやすい。

### 必要なテスト

- 高sigmaタスク直後に非並列タスクがある場合、その区間をバッファとして数えない
- 同じ締切余裕でも、実際の連続空き時間が長いplanを高く評価する
- `proposal.typ` のA/B例で A→B を選好する
- 合法的な並列タスクはバッファを遮らない

## Problem 3: 一日にタスクが集中する

### 現状

日ごとの作業負荷を評価する項目がない。一方で、次の要素はタスクの前倒しを促す。

- greedy initial solution が各タスクを可能な限り早く配置する
- deadlineより早く終わるほど早期完了ボーナスが付く
- 現在のbuffer評価も早い配置を選好する

その結果、締切まで数日の余裕があっても、今日または特定の一日に自由タスクが集中しやすい。

### 影響

形式上は実行可能でも、ユーザーが実際には従いにくい過密な日が生まれる。これは `proposal.typ` が問題としている「行動の拒否」を引き起こす。

### 改善方針

タスク数ではなく、日ごとの占有時間に凸ペナルティをかける。

最初の実装では、快適容量を超えた部分に二次ペナルティを与える。

```text
load(day) = その日にタスクが占有する時間
excess(day) = max(0, load(day) - comfortable_capacity(day))
daily_load_score = -W_DAILY_OVERLOAD × Σ excess(day)^2
```

必要であれば、快適容量以下でも緩い二次ペナルティを追加し、同じ総作業時間なら複数日に分散した配置を選好させる。

```text
daily_load_score =
    -W_DAILY_NORMAL × Σ load(day)^2
    -W_DAILY_OVERLOAD × Σ excess(day)^2
```

### 負荷の数え方

- 件数ではなく5分スロット数で測る
- 固定予定もその日の利用可能容量を減らすため負荷へ含める
- 合法的に重複する並列タスクは、単純に時間を二重加算しない
- 初期実装では区間のunionを日次負荷とする
- 将来的に認知負荷を表現する場合、通常タスク・parallel host・parallel guestへ異なる係数を持たせる

設定は `SleepConfig` から分離し、例えば次のような構造を検討する。

```rust
pub struct WorkloadConfig {
    pub comfortable_slots_per_day: i64,
    pub maximum_slots_per_day: i64,
}
```

`comfortable` 超過は緩いペナルティ、`maximum` 超過は強いペナルティとすることで、「多少忙しい日は許容するが、過密日は避ける」という性質を表現できる。

日ごとの分散だけを最小化すると、評価期間の端まで不必要に先送りする可能性があるため、締切評価と容量超過評価を組み合わせる。

### 必要なテスト

- 締切条件が同じなら、4時間を一日に置くより2時間ずつ二日に分けたplanを高く評価する
- 締切が近い場合は、必要に応じて一日へ集中できる
- 固定予定が多い日には自由タスクを追加しにくくなる
- 合法的な並列タスクを二重加算しない
- 快適容量以下の軽い日へ過剰なペナルティを与えない

## Problem 4: task abandonment が実装されていない

### 現状

`proposal.typ` では、全タスクが収まらない場合に次の順序で対応する。

1. 標準偏差の大きいタスクを、早く終わる可能性に期待して少し短く配置する
2. それでも収まらない場合、`abandonability` を考慮してタスクを諦める
3. 諦めたことをユーザーまたは音声アシスタントへ伝える

現在の実装は全タスクを常にスケジュールし、`abandonability` は deadline超過ペナルティを弱めるためだけに使う。`Task` のフィールド説明は「高いタスクからドロップされる」としている一方、`Plan` は全タスクを配置するとしており、公開APIの説明も矛盾している。

### 影響

実行不可能な量のタスクがある場合も、期限超過した予定としてすべて出力される。ユーザーやLLMは、実行予定なのか諦める候補なのかを区別できない。

### 改善方針

`Plan` が配置済みタスクと断念タスクを明示的に返すようにする。

```rust
pub struct Plan {
    pub schedules: Vec<(Point, Point, usize)>,
    pub abandoned_task_ids: Vec<usize>,
    pub warnings: Vec<PlanWarning>,
}
```

探索へdrop/restore近傍を追加し、次を評価する。

- タスクを残す inclusion benefit
- abandonabilityが低いタスクを落とす強いペナルティ
- deadline、sleep、daily workloadを破壊して全件を残すコスト

`abandonability = 1.0` でも無条件に落とすのではなく、収まるなら配置する。断念は、実行不可能または生活制約を大きく破る場合にだけ有利になる重み設計とする。

短縮については現在の `duration_score` とduration近傍を利用できるが、短縮されたタスクもユーザーへ通知できるよう、推定時間との差分をwarningとして返す。

### 必要なテスト

- 全件が収まる場合は断念しない
- 収まらない場合はabandonabilityの高いタスクを優先して断念する
- abandonabilityの低いタスクを守る
- 睡眠や日次最大容量を極端に破るより、断念を選ぶ
- abandoned taskと短縮taskが出力から判別できる

## Problem 5: 依存サイクルのタスクが黙って消える

### 現状

初期解のトポロジカルソートは、依存サイクルに含まれるノードを結果へ追加しない。`build_initial` はソート結果に含まれるタスクだけを配置するため、サイクル内のタスクはplanから消える。

その後の近傍操作は現在のplanを変形するだけであり、初期解から消えたタスクを復元できない。inclusion bonusも、存在しないタスクを探索へ再追加する機能にはならない。

### 影響

`takusu-core` を直接利用した場合、不正な依存入力がエラーにならず、「全タスクをスケジュールする」という契約に反して一部タスクが欠落する。

### 改善方針

plan実行前に依存グラフを検証し、不正入力を明示的なエラーにする。

- 依存先IDが存在する
- 自分自身へ依存していない
- 依存グラフがDAGである

公開APIは最終的に次の形を検討する。

```rust
pub fn plan(&self) -> Result<Plan, PlannerError>;
pub fn plan_partial(...) -> Result<Plan, PlannerError>;
pub fn plan_in_range(...) -> Result<Plan, PlannerError>;
```

既存呼び出し側への影響が大きい場合、まず `Planner::validate()` を追加し、serverから必ず呼ぶ段階を挟む。

### 必要なテスト

- 自己依存を拒否する
- 2ノード以上の循環依存を拒否する
- 存在しない依存IDを拒否する
- 正常なDAGでは従来どおり全タスクを配置する

## Problem 6: 睡眠時間を並列タスクごとに重複加算する

### 現状

`sleep_score` は、各スケジュールと睡眠窓の重複時間を単純加算している。同じ時間帯に合法的な並列タスクが複数ある場合、同じ実時間を複数回失ったものとして扱う。

### 影響

実際の睡眠損失より大きなペナルティが発生する。並列タスクが多いと `sleep_got` が負になり、二次ペナルティが極端に大きくなる可能性がある。

### 改善方針

各睡眠窓について、重なるスケジュール区間を切り出してマージし、区間unionの長さを `occupied` とする。

```text
intervals = sleep windowと各scheduleの共通部分
occupied = length(union(intervals))
sleep_got = sleep_length - occupied
```

この区間union処理は日次負荷評価でも再利用できるため、共通の内部utilityとして実装する。

### 必要なテスト

- 同時刻の並列タスク2件でも睡眠損失を1回だけ数える
- 重なりの一部だけが共通する複数区間のunionを正しく数える
- 重ならない区間は合計する
- `sleep_got` が0未満にならない

## 実装順序

### Phase 1: correctness

1. `plan_in_range` を元Planner + pinned partial solveへ変更する
2. 依存グラフ検証を追加する
3. 睡眠区間のunion計算を導入する

### Phase 2: schedule quality

4. 実際の連続空き時間に基づくbuffer評価へ変更する
5. 日次負荷評価と設定を追加する

bufferと日次負荷は双方とも時間区間の競合・空き時間を扱うため、共通表現を設計してから実装する。

### Phase 3: infeasibility reporting

6. abandonmentと短縮結果を `Plan` に表現する
7. server/API/LLMが断念・短縮・睡眠削減をユーザーへ提示できるようにする

## 検証方針

各問題の単体テストに加え、代表的な数日分のシナリオを固定fixtureとして用意する。

- 不確実な長時間タスクと短い確定タスク
- 固定予定が多い日と空いている日
- 数日後の締切を持つ複数タスク
- pinned予定を含む部分再計画
- 全件を収めると睡眠または日次最大容量を破るケース

評価関数の各項だけでなく、最終的なsolver出力について次の不変条件を検証する。

- task IDの重複・欠落がない。ただし明示的にabandonedへ分類されたtaskを除く
- 非並列タスクが重複しない
- 依存順序を満たすか、満たせない理由をwarning/errorとして返す
- fixed/pinned予定が移動しない
- 断念・短縮・生活制約違反が呼び出し側から判別できる
